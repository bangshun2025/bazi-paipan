## 二、问题本质

**表象**：搜索时弹窗高度抖动。

**根因链**：
```
弹窗无固定高度（仅 max-height: 80vh）
  → 高度由内容撑开
    → 搜索过滤减少条目 → 列表区内容高度骤降
      → flex 默认 min-height:auto 阻止收缩
        → 弹窗整体变矮 → 用户感知为"抖动"
```

这是典型的 **CSS Flexbox 布局陷阱**：`flex: 1` + 无明确容器高度 + `min-height: auto` 三者叠加，导致 `overflow-y: auto` 形同虚设。

---

## 三、修复方案

| 优先级 | 修改 | 代码 | 作用 |
|--------|------|------|------|
| P0 | `.archive-modal` | `height: 480px` | 锁定弹窗外框 |
| P0 | `.archive-modal-list` | `min-height: 0` | 解除 flex 默认约束，让 overflow 生效 |
| P0 | `.archive-modal-empty` | `display:flex; height:100%` | 空状态撑满不塌陷 |
| P1 | `oninput` → 防抖 | `setTimeout 150ms` | 减少输入中重绘次数 |

**改动量**：4 处，纯 CSS + 1 个 JS 辅助函数。不动核心逻辑。

## 五、经验总结

### 做得好的

1. **走正规流程**。用户最初想让 Leader 直接修，但要求走开发流程后，PRD→ADR→实现→验证的流水线保证了质量，13/13 一次过。
2. **根因分析到位**。产品经理没有停留在"加个固定高度"，而是把三层原因拆开（容器高度、flex 约束、空状态塌陷），每层独立修复。
3. **架构评审的价值**。架构师发现了 `min-height: 0` 是「最关键的一行代码」，并逐模块验证了 AI 弹窗、回收站、移动端的零影响。
4. **自动化验证**。测试师用 CDP + Python 脚本替代手工点击，13 条 AC 覆盖核心/边界/回归，可复现。

### 可改进的

1. **防抖优先级**。PRD 标 P2，架构师建议 P1，实际上实现后体验提升明显。产品经理对交互细节的优先级判断可以更积极。
2. **测试左移**。如果弹窗改造时就跑一遍搜索场景的自动化检查，这个 Bug 可能在上一轮就被发现，不需要用户反馈。

### 一条原则

> **CSS 弹窗类组件，必须锁定外框尺寸。** 弹窗 = 固定容器 + 内部滚动，这是 UI 基础范式。任何「由内容撑开高度」的弹窗设计，都隐含抖动风险。
